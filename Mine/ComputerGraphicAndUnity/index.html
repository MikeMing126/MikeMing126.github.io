前言

主要还是结合之前学校计算机图形学的课程、Game101课程和Unity相关知识，进行重新梳理。

计算机图形学

基础

点乘 叉乘

点乘 - 两向量间夹角，以及向量投影

  • 可以用于评估向量的方向、相似度（Close）等

叉乘 - 两向量的垂直向量，值为两向量构成四边形的面积

  • 注意左手坐标系用左手法则，右手坐标系用右手法则。
  • 用于判定向量左右/内外关系
  • 行列式计算叉乘

Unity 渲染管线

Unity Shader（零）-顶点着色器和片元着色器
猴子也能看懂的渲染管线（Render Pipeline）
GPU 渲染管线与着色器
图形渲染管线详细梳理

基础流程

CPU应用程序端渲染逻辑 - GPU渲染管线 - FrameBuffer

应用阶段 Application Stage

CPU主导

  • 元素剔除
    • GPU裁剪阶段也有剔除。但是在CPU阶段剔除可以减少传输的数据。
    • 依据FOV、FarClip、NearClip构成视锥体，并进行视锥体剔除
  • 排序
    • 基于RenderQueue渲染队列来进行排序处理，确保正确的渲染顺序并提高渲染效率。（深度测试/写入的开关、多个Pass时减少切换优化性能）
    • 半透明一般RenderQueue>不透明，不透明相同的RenderQueue则按照摄像机距离从前到后进行排序。半透明相同的RenderQueue则按照摄像机距离从后到前进行排序。
    • Camera.Depth（高的后渲染） > 材质类型(是否透明,具体由RQ判断) > RenderQueue > sorting layer > order in layer > 相机距离z。 sorting layer与order in layer适用于基于摄像机/世界空间的Canvas。以及Sprite
    • 参考文章Unity中SortingLayer、Order in Layer和RenderQueue的讲解 和 U3D sorting layer, sort order, order in layer, layer深入辨析
  • 数据的准备
    • 模型相关的: 顶点、法线、UV、切线、顶点色、索引列表
    • 灯光：灯光数据、材质信息
    • 变换矩阵：世界变换矩阵、依据摄像机构建的VP矩阵
  • 渲染状态的设置和绘制调用
    • SetPass Call
    • Draw Call

Vertex Shader 顶点着色器

  • 矩阵变换的计算
  • 计算光照公式生成逐顶点颜色（可选）
  • 生成/变换纹理坐标

Rasterization 光栅化

将三角形划分对应到单元格区域。
将图元变为片元。
3D游戏对应的片元大部分情况下会大于屏幕的宽和高。每个三角形都会计算片元并对应到屏幕坐标。

Fragment Shader 片元着色器

或称为Pixel Shader

  • 光照和颜色计算
  • 获取纹理，进行纹理映射
  • 往像素填充颜色值，处理混合以及特殊效果

Per-Fragment Operations 逐片元操作

在Fragment Shader到Frame Buffer之间。逐片元操作是可以配置但不可编程的，也就是说是由管线/硬件自身规定好的，我们只能对里边的内容进行配置。

进行测试与合并的工作。

  • 裁剪测试 Scissor Test
  • 透明度测试 Alpha Test （后面OpenGL改了不支持，相关逻辑可以放到Fragment Shader中去实现）
    • 提前设置一个透明度预值，只能实现不透明效果和全透明效果。
  • 模板测试 Stencil Test
  • 深度测试 Depth Test

然后进行Blending（透明度混合），并在进行一系列其他输出后，输出到帧缓冲区。

模板测试和深度测试

3.1 模板测试和深度测试

处理阶段都是位于逐片元阶段。

模板测试

referenceValue：当前模板缓冲片元的参考值
stencilBufferValue：模板缓冲区的值
comparisionFunc：比较函数

模板缓冲区：FrameBuffer 可以为屏幕上每一个像素点保留一个无符号整数值。

  • 最重要(用来比较的）两个值：当前模板缓冲区值（StencilBufferValue）、模板参考值（ReferenceValue）
  • 模板测试主要就是对这两个值进行特定的比较操作，例如Never、Always、Equal等，具体参考上文的表格
  • 模板测试后，要对模板缓冲区的值进行更新操作，例如Keep，Replace等，具体参考上文表格
  • 模板测试之后，可以通过不同的结果对模板缓冲区做不同的更新操作，例如模板测试成功的操作Pass、模板测试失败的操作Fail、深度测试失败的操作ZFail、还有正对正面和背面更新操作Passback，Passfront，Failback等…

应用

描边

深度测试

处理物体的遮挡关系。

总体上还是和模板测试比较类似的。

作用

  • 控制渲染顺序
    • 画家算法：从远处开始画，近处的东西叠加在上面（存在多画无用的问题）
    • ZBuffer算法：通过深度缓冲区来控制渲染顺序
  • 控制Z-Buffer对深度的存储
    • 什么时候更新、什么时候使用深度缓冲区。Z Test和Z Write。
  • 控制不同类型物体的渲染顺序
    • 透明物体
    • 不透明物体
    • 渲染队列
  • 减少Overdraw
    • Early-Z
    • Z-cull
    • Z-check

ZBuffer

深度缓冲区 Z-Buffer
颜色缓冲区：最终显示的区域。

Z Write：
开启情况下，深度测试通过会写入深度缓冲区。（不开启时，通过时，不写入深度缓冲区，但仍然会写入颜色缓冲区）。
默认情况下为开启，且比较为 ZTest LEqual（一开始初始值为无限大）。

渲染队列

按照渲染顺序从先到后排序，队列数越小，越先渲染；反之同理。
Unity中设置渲染队列：

  • 语法：Tags { “Queue” = “Transparent”}
  • 默认是Geometry

Unity中不透明物体的渲染顺序：从前往后，也就是说深度小的先渲染，其次再渲染深度大的
Unity中透明物体的渲染顺序：从后往前（类似画家算法，会造成OverDraw）

Early-Z

传统的渲染管线中，ZTest是在Blending阶段，这时进行深度测试的话，所以对象的像素着色器都会计算一遍，没有性能提升，只是为了得到正确的效果，造成了大量的无用计算。（深度测试失败的片元是已经经过计算的，到这一步不通过被抛弃，那么前边的计算就是无用功了）

为了减少这些不必要的计算，现代GPU运用了Early-Z技术，在顶点和片元阶段之间（光栅化之后，片元着色器之前）进行一次深度测试。

如果这次深度测试失败，那就不用在片元着色器中作无关紧要的计算了，这样一来就会带来性能提升。

最终的ZTest仍然要进行，以保证正确的遮挡关系。

深度值

坐标在经过MVP矩阵变换后，会获得一个深度值。
其深度值为z/w

应用

  • 透视物体（使用ZTest - Greater）
  • shadow map阴影贴图实现原理
