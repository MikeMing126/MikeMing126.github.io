计算机进程

定义：一个具有一定独立功能的程序关于某个数据集合的一次活动。

进程是资源分配的基本单位。
资源包括：

  1. CPU时间
  2. 内存[分配独立的虚拟地址空间]
  3. 输入/输出设备
  4. 网络连接（如TPC/IP socket）
  5. 信号量和锁 - 用于进程间通信和同步，管理共享资源的访问
  6. 其他进程间通信机制，如管道、消息队列、共享内存等。

执行模式：

  1. 顺序执行Sequential Execution
    特性：Sequential(顺序性)、Close(封闭性，独占所有资源)、Reappear（可重现 结果只与初始条件有关）
  2. 并发执行 Concurrency Execution
    没有以上特性。

由Running进入Blocked主要是因为等待事件。如IO请求、访问临界资源等。

线程

一个进程可以拥有多个线程，共享进程资源。
线程之间可以并发运行。

把进程作为资源分配单位和调度单位这两个属性分开处理，即进程还是作为资源分配的基本单位，但是不作为调度的基本单位（很少调度或切换），把调度执行与切换的责任交给线程，即线程成为独立调度的基本单位，它比进程更容易（更快）创建，也更容易撤销。

不过，一个线程的崩溃会导致其他的线程也崩溃。

资源

同进程内的多线程共享进程的资源。[代码段、数据段、打开的文件等。]，此外，每个线程还有一些额外的数据：

额外资源，由此确保线程的控制流是相对独立的。

  1. PC Program Counter 程序计数器
  2. Register Set 寄存器集合
  3. Stack 栈

同一个进程的线程切换时，也只需要切换线程的私有数据，开销要小很多。

临界资源 和 临界区

Critical Resource & Critical Region(Section)
定义：
临界资源： 一次仅允许一个进程访问。[Software & Hardware]
临界区:访问临界资源的程序区

临界区4条件:

  1. 不能有两个进程同时处于临界区
  2. 不对CPU的数量和速度做任何假设
  3. 临界区外的进程不得阻塞其他进程
  4. 不得使进程无期限地等待进入临界区

通信方案

  1. 匿名管道(pipe)
    • 实质是内核缓冲区 一端写入一端读出[单向数据流]。
    • 匿名，仅能用于父子进程间通信
    • 无格式字节流
  2. 有名管道(FIFO)
    • 和匿名管道有些类似，但不限于父子进程。
    • 名字存于文件系统中，内容存放再内存中
  3. 消息(Message)队列
    • 本质是存放在内存中的消息的链表，而消息本质上是用户自定义的数据结构。
    • 允许一个或多个进程向它写入或读取消息，读取某个消息后消息从消息队列中删除
    • 可以实现消息的随机查询,消息不一定要以先进先出的次序读取,也可以按消息的类型读取.比FIFO更有优势。
    • 用户进程写入数据到内存中的消息队列时，会发生从用户态拷贝数据到内核态的过程；同样的，另一个用户进程读取内存中的消息数据时，会发生从内核态拷贝数据到用户态的过程。因此，如果数据量较大，使用消息队列就会造成频繁的系统调用，也就是需要消耗更多的时间以便内核介入。
  4. 共享内存
    • 对于共享内存机制来说，仅在建立共享内存区域时需要系统调用，一旦建立共享内存，所有的访问都可作为常规内存访问，无需借助内核。
  5. 信号量（Semaphores）和PV(DownUp)
    • 消息队列无需避免冲突，而共享内存机制可能会发生冲突。
    • 保证共享内存在任何时刻仅有一个进程在访问，并且使得进程可以按照某个特定顺序访问共享内存，使用信号量来实现进程的同步和互斥机制。
    • P(Down) 信号量-1，表示申请占用一个资源
    • V(Up) 信号量+1，表示释放一个资源
    • 当信号量的值为 2 的时候，表示有 2 个资源可以使用，当信号量的值为 -2 的时候，表示有两个进程正在等待使用这个资源。
    • 实现互斥 - 初始化信号量为1，同个进程进行PV
    • 实现同步 - 不同进程进行PV
  6. 信号
    • 唯一的异步通信机制，它可以在任何时候发送信号给某个进程。通过发送指定信号来通知进程某个异步事件的发送，以迫使进程执行信号处理程序。信号处理完毕后，被中断进程将恢复执行。用户、内核和进程都能生成和发送信号。
    • 如果该进程当前并未处于执行状态，则该信号就有内核保存起来，知道该进程回复执行并传递给它为止。如果一个信号被进程设置为阻塞，则该信号的传递被延迟，直到其阻塞被取消是才被传递给进程。
  7. Socket 套接字
    • 支持跨网络
    • 本质其实是一个编程接口（API），是应用层与 TCP/IP 协议族通信的中间软件抽象层，它对 TCP/IP 进行了封装。它把复杂的 TCP/IP 协议族隐藏在 Socket 接口后面。对用户来说，只要通过一组简单的 API 就可以实现网络的连接。

乐观锁和悲观锁

乐观锁与悲观锁

悲观锁：
悲观锁基于”最坏情况”假设，认为在数据处理过程中必然会出现并发冲突，因此在操作数据前就对其加锁，确保同一时间只有一个线程能访问该数据。

乐观锁：
乐观锁假设并发冲突发生的概率较低，不主动加锁，而是在更新数据时检查是否有其他线程修改过该数据。若数据未被修改，则执行更新；若已被修改，则放弃或重试。

互斥锁 Mutex

提供互斥访问，一次只有一个线程可以获得锁。
通过阻塞保证互斥性。

应用：适用于保护共享资源，防止并发访问造成数据混乱，例如更新全局变量、文件写入等。

通常用于实现临界区。进入临界区之前会尝试获取互斥锁，如果成功（即锁未被其他线程持有），则进入临界区。当线程退出临界区时，它会释放互斥锁，使得其他等待该互斥锁的线程可以获取锁并进入临界区。

读写锁 Read-Write Lock

允许多个读线程同时访问资源，但在写线程访问资源时，所有其他线程（无论是读线程还是写线程）都不能访问资源。

读的时候也会获取读锁。并在处理完毕后释放读锁。
写的时候首先检查是否有其他锁（包括读/写锁）

C++实现

C++17引入的std::shared_mutex。
在写的时候用std::unique_lock获取写锁，读的时候用std::shared_lock获取读锁。

自旋锁 Spinlock

是一种特殊类型的锁，当线程无法立即获取锁时，它不会立即进入阻塞状态，而是在一个循环中不断地尝试获取锁，直到成功为止。这种锁适用于锁持有时间短且线程不希望在重新调度上花费过多时间的情况。

在获取不到锁时，自旋等待并再次尝试获取锁。

可以避免线程在等待锁时进入阻塞状态，从而减少了线程调度的开销。

自旋锁应该尽量只在持锁时间非常短的情况下使用，否则会由于忙等待导致CPU资源的浪费。

死锁问题 Deadlock

两个或多个线程或进程因争夺资源而相互等待，导致所有参与者都无法继续执行的状态。

死锁的发生需要满足以下四个条件，这些条件同时成立时，系统可能进入死锁状态：

  1. 互斥
    至少有一个资源是非共享的，某一时刻只能被一个线程或进程占用。
  2. 占有和等待
    线程已经持有资源，同时又请求新的资源，但未释放已有资源。
  3. 不可抢占（不可剥夺）
    已被分配的资源不能强制剥夺，只能由持有该资源的线程或进程主动释放。
  4. 循环等待
    存在一个线程或进程的循环等待链，链中的每个线程或进程都在等待下一个线程持有的资源。

  • 锁争用undefined多线程频繁请求同一资源导致锁竞争激烈，线程阻塞等待锁释放，降低系统吞吐量。

优化策略
  • 仅对临界区进行加锁。
  • 读写锁优化读多写少的场景
  • 使用无锁数据结构
    锁的等待和上下文切换增加了开销。使用无锁数据结构（如 std::atomic 或自定义的无锁队列）替代加锁。
  • 避免长时间持有锁
  • 使用尝试加锁或超时锁
    使用 std::try_lock 或 std::unique_lock 的超时功能避免长时间阻塞。
  • 使用分区锁。将数据划分为多个分区，每个分区使用独立的锁
  • 替代同步机制
    用条件变量、信号量、事件驱动模型等替代锁。

实现区别
  1. lock_guard 是基于互斥锁 std::mutex 实现的，unique_lock 是基于通用锁 std::unique_lock 实现的，unique_lock 可以实现比 lock_guard 更灵活的锁操作。
  2. lock_guard 是不可移动的（moveable），即不能拷贝、赋值、移动，只能通过构造函数初始化和析构函数销毁，unique_lock 是可移动的，可以拷贝、赋值、移动。
    unique_lock 提供了更多的控制锁的行为，比如锁超时、不锁定、条件变量等。
  3. unique_lock 比 lock_guard 更重，因为它有更多的功能，更多的开销。如果只需要简单的互斥保护，使用 lock_guard 更好。
  4. unique_lock 支持手动解锁，而 lock_guard 不支持。

原子操作

深入探究C++原子操作：原理、应用与性能权衡

原子操作（Atomic Operation），确保对共享数据的访问是不可分割的，即在多线程环境下，原子操作要么完全执行，要么完全不执行，不会出现中间状态。

C++中的无锁编程，常依赖于std::atomic类型和相关操作。
优势：

  • 确保数据一致性的同时，性能提升(减少阻塞和上下文切换)
  • 避免死锁和饥饿

原子操作和互斥锁都是同步机制，但它们有不同的适用场景和性能特点。原子操作通常用于简单的同步场景，如计数器和标志位，而互斥锁则适用于更复杂的同步场景，如保护共享数据结构。

原子操作的内存模型
为什么要考虑内存顺序？

在多线程程序中，不同线程对内存的读写操作可能导致意想不到的结果，这主要是因为现代计算机系统和编译器通常会对操作进行重排序，以优化性能和资源利用率。然而，这种重排序可能会打破代码的逻辑顺序，导致数据竞争和一致性问题。

内存顺序（Memory Ordering），是一种规则，它定义了操作的可见性和执行顺序，是确保多线程程序正确性的关键。（防止单线程指令的合理重排在多线程的环境下出现顺序上的错误。）在C++中，这些规则通过原子操作的内存顺序标志来实现。

内存顺序

C++定义了多种内存模型，用于控制操作的内存可见性和顺序。这些内存模型包括：

  • memory_order_seq_cst：顺序一致性，默认的内存顺序，提供最强的顺序保证。
  • memory_order_relaxed：不对操作的内存顺序做任何保证。
  • memory_order_acquire：确保在此操作之前的所有内存写入都对其他线程可见。用于读取操作，保证在该操作之后的内存读写不会被重排序到该操作之前。它确保对共享数据的读取操作可以看到之前的写入操作的结果。
  • memory_order_release：用于写入操作，保证在该操作之前的内存读写不会被重排序到该操作之后。它确保写入操作对后续的读取操作可见。
  • memory_order_acq_rel：同时具有memory_order_acquire和memory_order_release的特性。
    选择合适的内存模型对于保证程序的正确性和性能至关重要。

由于内存空间都是预先划分好的，也就不会像分段会产生间隙非常小的内存，这正是分段会产生内存碎片的原因。而采用了分页，那么释放的内存都是以页为单位释放的，也就不会产生无法给进程使用的小内存。

内存碎片 内部碎片和外部碎片

内部碎片

内部碎片是已经被分配出去（能明确指出属于哪个进程）。由于被装入的数据块小于分区大小，从而导致分区内部有空间浪费，这种现象成为内部碎片。

内部碎片是处于区域内部或页面内部的存储块。占有这些区域或页面的进程并不使用这个存储块。而在进程占有这块存储块时，系统无法利用它。直到进程释放它，或进程结束时，系统才有可能利用这个存储块。

为了有效的利用内存，使内存产生更少的碎片，要对内存分页，内存以页为单位来使用，最后一页往往装不满，于是形成了内部碎片。

因为所有的内存分配必须起始于可被 4、8 或 16 整除（视处理器体系结构而定）的地址或者因为MMU(Memory Management Unit)的分页机制的限制，决定内存分配算法仅能把预定大小的内存块分配给客户。假设当某个客户请求一个43字节的内存块时，因为没有适合大小的内存，所以它可能会获得 44字节、48字节等稍大一点的字节，因此由所需大小四舍五入而产生的多余空间就叫内部碎片。

外部碎片

外部碎片是指未被分配的空闲内存区域分散且不连续，导致无法满足较大内存请求的现象。即使空闲内存的总量足够，但由于分散成多个小块，无法合并使用。

采用分段管理的虚拟内存会产生外部碎片。

内存对齐

很详细的一篇文章：
内存对齐：程序员必知的性能优化秘籍

原因

  1. 平台原因(移植原因)：不是所有的硬件平台都能访问任意地址上的任意数据的；某些硬件平台只能在某些地址处取某些特定类型的数据，否则抛出硬件异常。比如处理器通常只能从地址为 8 的倍数的内存位置开始读取数据。
  2. 性能原因：数据结构(尤其是栈)应该尽可能地在自然边界上对齐。原因在于，为了访问未对齐的内存，处理器需要作两次内存访问；而对齐的内存访问仅需要一次访问。

说明

数据在内存中的起始地址必须是其对齐值的整数倍。

对齐系数（Alignment Factor）：
这是编译器默认的对齐基准值，在 64 位系统中通常为 8 字节；
可以通过 #pragma pack(n) 指令进行修改，其中 n 可以是 1、2、4、8、16；

有效对齐值（Effective Alignment）：
这是实际决定成员对齐要求的关键值，计算公式：min(数据类型大小, 对齐系数)；
例如：在 64 位系统中，int 类型的有效对齐值为 min(4, 8) = 4，double 类型的有效对齐值为 min(8, 8) = 8；

内存对齐分为两个层面：内对齐（Internal Alignment）和外对齐（External Alignment），它们共同确保数据访问的高效性和数据在内存中的正确布局。

内对齐规则：

  • 结构体第一个成员的偏移量始终为 0；
  • 后续每个成员的偏移量必须是该成员(自己)有效对齐值的整数倍；
  • 如果自然偏移量不满足对齐要求，编译器会在前一个成员后插入填充字节（Padding Bytes）；
  • 这些填充字节不存储任何有效数据，但它们对于保证内存对齐和程序正确运行起着至关重要的作用；
  • 填充字节确保每个成员都从其有效对齐值的整数倍地址开始，从而保证 CPU 能够高效地访问数据；

外对齐规则：

  • 结构体的总大小必须是结构体中最大成员的有效对齐值的整数倍；
  • 如果不满足这个要求，编译器会在最后一个成员的尾部添加填充字节；

当结构体中包含嵌套结构体时，嵌套结构体的起始地址必须满足其自身的对齐要求。嵌套结构体的对齐要求由其内部最大成员的有效对齐值决定。

在计算嵌套结构体的外对齐时，不是按照嵌套结构体本身的大小来计算，而是按照结构体中所有基本数据成员类型的最大有效对齐值来计算。

位域

当结构体内包含位域时，内存对齐规则会变得更加复杂。位域的内存对齐遵循以下核心原则：

  1. 容器类型对齐： 位域总是被打包到其声明的基本类型（如 int、unsigned int、char、short 等）中，这个基本类型称为位域的 “容器类型”。容器类型的起始地址必须遵守其自身的自然对齐规则。

  2. 位域填充顺序： 编译器将位域填充到容器类型中时，可能从低位到高位填充（小端系统常见），也可能从高位到低位填充（大端系统常见）。具体的填充顺序取决于编译器和目标平台的字节序。

  3. 容器边界限制： 位域不能跨越其容器类型的边界。如果当前容器中剩余位数不足以容纳下一个位域，编译器会跳过剩余位，为下一个位域分配新的容器实例。

  4. 结构体整体对齐： 包含位域的结构体，其整体对齐要求由所有成员（包括位域的容器类型）中的最大对齐值决定，这与普通结构体的对齐规则相同。

零宽度位域 Zero-width bit-field

零宽度位域（:0）是一个特殊的编译器指令，当编译器遇到它时，即使前面还有剩余空间，编译器也会强制跳过这些剩余位，将下一个位域或普通成员从下一个容器类型的边界开始。它本身并不占用存储空间，但会影响后续成员的内存布局。比如，以下代码将会强制使 status 位域成员从新的 unsigned int 边界开始。

优化建议

在实际开发中，合理利用内存对齐可以显著提升程序性能。以下是一些优化建议：

  1. 按大小降序排列：将较大的数据类型放在前面，较小的放在后面，可以有效减少填充字节；
  2. 相关数据聚集：将经常一起访问的成员放在相邻位置，提高缓存命中率；
    避免跨缓存行：将关键数据结构按缓存行大小对齐，确保不跨越缓存行边界；
  3. 位域的有效使用：使用位域将多个小字段打包到同一字节，节省内存；
  4. 编译器优化指令：合理使用 #pragma pack 和 __attribute__((aligned)) 等编译器特定指令；
  5. 避免伪共享：在多线程环境中，将不同线程访问的数据放在不同的缓存行中，具体做法是将数据结构按缓存行大小对齐，避免不同线程访问的数据在同一缓存行中；
    通过合理的内存对齐优化，可以在不增加算法复杂度的情况下，获得显著的性能提升。但需要注意的是，过度优化可能导致代码可读性下降，需要在性能和可维护性之间找到平衡。
