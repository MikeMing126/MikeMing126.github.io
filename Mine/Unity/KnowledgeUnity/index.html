The Unity engine is built with native C/C++ internally, however it has a C# wrapper that you use to interact with it.

分层架构

本身底层是C++。上层交互用C#。

关键设计原则：

  • 模块解耦：每个子系统（如Rendering/Physics）以DLL形式动态加载
  • 接口抽象：通过IScriptableRenderPipeline等接口实现可替换组件
  • 数据驱动：YAML格式的Scene/Asset序列化机制

多平台

硬件抽象层，定义对于的结构。

关键机制：

  • 平台定义文件（PlatformDefines）：通过宏指令隔离平台相关代码
  • IL2CPP转换层：将C# IL代码转换为C++，作为AOT的实现方案。
  • UnityPlayer动态库：封装各平台原生API调用

GC

对Mono和IL2CPP后端均使用Boehm垃圾收集器。默认使用增量GC。

C#反射开销

Mono和IL2CPP内部缓存所有C#反射 (System.Reflection) 对象，且Unity不会对它们进行垃圾回收。GC会扫描反射对象而导致不必要的开销。

减少这种开销，避免使用Assembly.GetTypes 和 Type.GetMethods() 等会在运行时创建许多C#反射对象的方法。

安全性系统

以Job使用主线程的数据引用为例子：
C# Job System通过给每一个需要操作数据的Job一份数据的拷贝而不是主线程中数据的引用来避免这个问题。拷贝和原本的数据独立，从而排除了资源竞争。

拷贝数据的方式表明了一个Job只能访问可以位块传输的数据类型(blitable data types)。这种数据类型在托管代码和原生代码之间进行传递的时候不需要类型转换。

NativeContainer

安全性系统中拷贝数据的缺点是单个job的计算结果是与外部隔离的。为了突破这个限制，我们需要把结果放在一种共享内存——NativeContainer中。

NativeContainer是一种托管的数据类型，为原生内存提供一种相对安全的C#封装。它包括一个指向非托管分配内存的指针。当和Unity C# Job System一起使用时，一个NativeContainer使得一个Job可以访问和主线程共享的数据，而不是在一份拷贝数据上工作。

DrawCall

关于DrawCall，可以结合计算机图形学的渲染管线做基础了解。

DrawCall的性能问题:
在调用DrawCall之前，CPU都要向GPU发送很多内容，比如数据，状态，命令等。CPU完成了准备工作，提交了命令，GPU才可以开始工作。GPU渲染能力很强，渲染100个三角形网格和10000个三角形网格，区别不大。所以，GPU渲染速度往往优于CPU提交命令的速度。

手动设置渲染参数很繁琐。这就是游戏开发人员引入材质概念的原因。材质是一种包含有关如何绘制对象的信息的数据结构。它包含一个着色器及其所有参数，以及有关如何设置GPU渲染状态的信息。

针对Drawcall我们主要的思路就是每个物体尽量减少渲染次数，多个物体最好一起渲染，或者叫，批处理（Batching）。

Unity SetPasses:
SetPasses描述了一种更昂贵的操作：材质更改。更改材质很昂贵，因为我们必须设置一个新的渲染状态。其中包括着色器参数和管线设置，例如Alpha Blending，Z-Test，Z-Writing。

批处理 Batching

使用同一种材质的物体可以进行批处理。
其纹理、顶点着色器、片元着色器的代码都一致。可以把顶点数据合并成一个，仅调用一次DrawCall。

Unity静态批处理

静态批处理，不限制模型的顶点数量，它只在运行开始阶段，它只在运动开始阶段，合并一次模型。这要求这些物理不能被移动！

优点是，合并制作一次，后面渲染不用合并，节约性能。
缺点是，可能占用较多内存。比如，没用静态处理，一些物体共享了网格，比如地图有100个树木，共享一个模型。如果把这些变成静态批处理，需要100倍的顶点数据内存。

Unity动态批处理

动态批处理，则每一帧渲染，都需要合并一次模型网格。

优点是，这些物体可以被移动。
缺点是，因为每次要合并，需要有一些限制，比如，顶点数目不能超过300（不同Unity版本不一样）。

在游戏运行过程中重复绘制同一像素的问题。

  1. 透明OverDraw的减少：
    由透明度引起的OverDraw对性能特别不利，因为渲染通常伴随Alpha Blend，这需要额外的读取和算术运算。
    更糟糕的是，透明材质通常不会写入Z-buffer，因此最终会写入更多无法丢弃的像素。

    • 减少绘制的透明像素的数量及其成本。

  2. 不透明OverDraw的减少：
    Unity会尽最大努力从近到远渲染不透明元素。这就是说，我们首先渲染距离摄像机更近的元素，以便使用Z-Test剔除被遮挡的物体的像素。
    该排序基于相机与每个对象的边界框中心之间的距离，非常有利于GPU性能。
    但是，在某些情况下，会导致渲染队列排序混乱，这会使OverDraw增加。

    • 天空球 中心点和相机位置关系
    • 将对象拆分为较小的子对象，以便Unity可以获取更准确的原点来更有效地对绘图调用进行排序。
    • Batching 导致的OverDraw

一篇比较详细的文章:
Unity 协程(Coroutine)原理与用法详解

线程、进程、协程的区别

进程拥有自己独立的堆和栈，既不共享堆，亦不共享栈，进程由操作系统调度。

线程拥有自己独立的栈和共享的堆，共享堆，不共享栈，线程亦由操作系统调度(标准线程是的)。[这里说的时KLT]

协程和线程一样共享堆，不共享栈，协程由程序员在协程的代码里显示调度 避免了无意义的调度，但失去了标准线程使用多CPU的能力。

协程的意义

非抢占式的多任务。将任务分到多个帧中去执行。
游戏中逻辑更新和画面更新的时间点有确定性，必须按照帧序列严格保持同步，否则就会出现游戏中对象不同步的现象。虽然多线程也能实现对象同步等效果，但是对一个大型游戏而言，游戏中的对象非常多，用多线程来实现很容易出Bug，而且往往很难找到Bug在哪，这无疑会加大开发的难度。

为了开发便利，统一生命周期管理，避免多线程锁的问题，Unity采用单线程逻辑。

为此，UnityEngine对其API和Componet等做出了限制，不能在主线程之外的线程中使用它们，这就确保了它们始终是在单线程中运行。

但是，对于一些耗时的操作（即需要较长时间才能执行完的代码块），我们不能一直等待其执行完毕再执行接下来的操作，否则游戏会出现卡顿等现象。例如，在游戏中等待1s再去执行某个操作时，如果我们让主线程等待1s，那么游戏画面就会停顿1s，这是不行的。为了解决这类问题，就需要使用协程——在主线程运行的同时开启另一段逻辑处理，来协同当前程序的执行。

显然，协程的出现是为了解决单线程的问题，如同线程的出现是为了解决进程的问题一样。我们可以想象，如果程序越来越复杂，进程之上还会有xx程，协程之下还会有xx程。

携程的执行时间

在调用携程(StartCoroutine)时会立刻执行对应的逻辑处理。
在yield后，携程停止执行，并返回执行下一行代码。

大部分协程，都是在Update和LateUpdate之间进行处理。 具体的可以参考官方API Documentation。
几个例外：

  • yield return new WaitForFixedUpdate(); //FixedUpdate和Update之间
  • yield return new WaitForEndOfFrame(); //在LateUpdate之后

yield后如果还有逻辑需要处理，其会在下一帧的Update和LateUpdate之间进行处理。(以yield return null为例子)

  • public void StopCoroutine(Coroutine routine)

对于每个协程，Unity 调用迭代器的 MoveNext() 方法。

如果 MoveNext() 返回 true，Unity 会查看 Current 属性（即 yield return 后面的对象）。

Unity 根据 Current 的类型（是 null、WaitForSeconds 还是其他）来判断这个协程是否满足了继续执行的条件。

GC的策略

  • Mark-Sweep。Tracing garbage collection。通过一个RootObject，遍历该对象的引用变量，并且标记。递归该过程确认所有reachable对象，剩余为garbage。(Lua)
  • 引用计数Reference Counting。当引用为0时，则该对象为garbage。
    存在问题：循环引用、计数空间开销、修改引用数目带来的速度开销以及原子性要求、非实时（一个引用的变化可能递归得导致一系列引用修改，内存释放)

基于Mono的UnityGC

Unity的脚本后端是基于Mono的实现（当然现在多了个IL2CPP，不过也是类似的GC实现)，而Mono使用的GC是所谓的Boehm–Demers–Weiser garbage collector。是Mark-Sweep 的实现，它会在需要进行GC时占用主线程，进行遍历-标记-垃圾回收的过程，然后在归还主线程控制权。

  • 非压缩式的，所以卡顿现象相对较轻，但是对内存利用率进一步下降了，会有内存碎片的问题
