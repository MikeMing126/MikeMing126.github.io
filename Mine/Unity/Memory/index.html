前话

CPU访问内存是缓慢的。Cache命中率很重要。
这也是Unity的DOTS的目标。

注意移动设备的区别

  • 没有独立显存(爆内存时也要考虑显存的使用)
  • 缓存更小

了解安卓的Low Memory Killer(LMK)

  • Native 系统内核，例如adbd
  • System 系统级应用，例如system server
  • Persistent 用户级应用，例如电话，蓝牙，wifi
  • Foreground 前台应用，当前正在使用的Activity
  • Perceptible 辅助类应用，例如搜索，音乐，键盘
  • Service 一些驻后台的线程服务，例如云服务，垃圾回收
  • Home 桌面
  • Previous 上一个使用的应用
  • Cached 后台应用

内存指标

  • Resident Set Size (RSS) [当前应用所应用掉的所有内存]
  • Proportional Set Size (PSS)[当前应用+共用服务的均分]
  • Unique Set Size (USS)[只包含自己的，不包含公共Service]

Unity中使用的内存

Unity本身是C++编写的。是一个C++引擎。其内存主要分为三部分：

C++
Bindings.cs（C++和C#绑定在一起。）
C#

VM一直存在。il2cpp也是VM。主要用于跨平台

  • Native Memory:C++ memory that Unity uses to run the engine.项目资源、UnityNative 子系统。一般不能直接通过C#访问。
  • Managed Memory:托管内存，托管堆和垃圾回收器，自动分配和指定的内存
  • C# Unmanaged Memory：C#非托管内存。需要手动分配和释放。Unity内可以通过Unity.Collections命名空间、Collections package、UnsafeUtility.Malloc和UnsafeUtility.Free进行使用。

注意：Editor和Runtime是不同的。包括内存的分配方式。

从使用上来说，分为引擎管理的内存和用户管理的内存。

UnityProfile监测不到的内存：

  • 用户分配的Native内存[Native插件、Lua(Lua自己管理)]

Unity Native Memory

什么影响因素较大？

  • Scene中，大量的GameObject （因为底层C++要分配内存）
  • Audio
    • DPS buffer, 优化和CPU的交互，但是也会有声音延迟的问题。
    • Force to mono 解决双声道的双倍内存问题
  • Code Size（不仅是影响编译，还影响内存）
    • 模板泛型的乱用
  • AseetBundle
  • Resources文件夹
    • 官方建议是不要用Resources了，用AB
  • Textrue
    • upload buffer 类似于Audio。
    • r/w 能不开不开，不然会多存
    • Mipmaps
  • Mesh
  • Assets 参考官网资源管理的Best practice

Unity Managed Memory

  • VM内存池(Mono)
  • GC机制
    • Boehm机制 —— 没有分代，也没有压缩
    • 下一代GC：渐进式GC、IL2CPP（IL2CPP和Boehm还有不同）
    • 内存碎片化问题，建议先操作大内存，再操作小内存
    • 僵尸内存[活跃度不高、没有被释放]

最佳实践：

  • Dont Null it，Destroy it.
  • Class VS Struct
  • Pool in Pool
  • Closures and anonymous methods
  • Coroutines(其实也算上面匿名的例子) 用的时候生产，不用的时候销毁。
  • Configurations 配置表
  • Singleton 勿乱用
